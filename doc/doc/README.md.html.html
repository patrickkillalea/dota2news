  <h1>
    <a href="#%20mean%20stack" name="%20mean%20stack" class="pilcrow">&#182;</a>
    <a href="http://mean.io/"><img src="http://mean.io/system/assets/img/logos/meanlogo.png" alt="MEAN Logo" title="" /></a> MEAN Stack
  </h1>
</div>


<p>MEAN is a framework for an easy starting point with <a href="http://www.mongodb.org/">MongoDB</a>, <a href="http://www.nodejs.org/">Node.js</a>, <a href="http://expressjs.com/">Express</a>, and <a href="http://angularjs.org/">AngularJS</a> based applications. It is designed to give you a quick and organized way to start developing MEAN based web apps with useful modules like Mongoose and Passport pre-bundled and configured. We mainly try to take care of the connection points between existing popular frameworks and solve common integration problems.</p>

<li>install <a href="http://www.microsoft.com/en-us/download/details.aspx?id=34673">Microsoft Visual Studio C++ 2012 Express</a></li>

</ol>


<div class="highlight"><pre><code><span class="nv">$ </span>npm update -g
</code></pre></div>

    Overview
  </h3>

<p>The MEAN CLI is a simple Command Line Interface for installing and managing MEAN applications. As a core module of the Mean.io project, it provides a number of useful tools to make interaction with your MEAN application easier, with features such as: scaffolding, module creation and admin, status checks, and user management.</p>


<div class="highlight"><pre><code><span class="nv">$ </span>mean
<span class="nv">$ </span>mean --help
<span class="nv">$ </span>mean <span class="nb">help</span>
</code></pre></div>



<p><code>mean help</code> can also be used in conjunction with any command to get more information about that particular functionality. For example, try <code>mean help init</code> to see the options for init</p>


<div class="highlight"><pre><code><span class="nv">$ </span>mean <span class="nb">help</span> <span class="o">[</span><span class="nb">command</span><span class="o">]</span>
</code></pre></div>




<div class="pilwrap" id="users">
  <h3>
    <a href="#users" name="users" class="pilcrow">&#182;</a>
    Users
  </h3>
</div>


<p><p>Information can be display for a specific customer via <code>mean user email</code>. Email is required. User roles can be assigned or removed with the <code>--addRole (or -a)</code> and <code>--removeRole (or -r)</code> options, respectively.</p>
  <p>For example, the <i>admin</i> role is required to edit tokens.</p></p>


<div class="highlight"><pre><code><span class="nv">$ </span>mean user &lt;email&gt;
<span class="nv">$ </span>mean user &lt;email&gt; --addRole &lt;role&gt;<span class="p">;</span>
<span class="nv">$ </span>mean user &lt;email&gt; --removeRole &lt;role&gt;<span class="p">;</span>
</code></pre></div>

  </h4>
</div>


<p>To create a new MEAN app, run <code>mean init</code>. Name for the application is optional. If no name is provided, "mean" is used. The MEAN project will be cloned from GitHub into a directory of the application name.</p>


<div class="highlight"><pre><code><span class="nv">$ </span>mean init <span class="o">[</span>name<span class="o">]</span>

  <h2>
    <a href="#packages_1" name="packages_1" class="pilcrow">&#182;</a>
    Packages
  </h2>
</div>


<p>Everything in mean.io is a package and when extending mean with custom functionality make sure you create your own package and do not alter the core packages.</p>

<p>The mean.io package system allows developers to create modular code that provides useful tools that other mean developers can use. The packages, when published, are plug-and-play and are used in a way very similar to traditional npm packages.</p>

<p>The mean.io package system integrates all the packages into the mean project as if the code was part of mean itself and provides the developers with all the necessary tools required to integrate their package into the host project.</p>

<p>There are two types of packages:</p>

<p><strong>Custom Packages</strong> are generated by the mean scaffolder and contain most of your application logic. Custom packages are found in <em>/packages/custom</em> and can be published as a contrib package for use by other developers.</p>

<p><strong>Contrib Packages</strong> are installed by the mean installer and are found at <em>/packages/contrib</em>. Contrib packages are "plug and play".</p>

<span class="p">});</span>
</code></pre></div>



<p>MEAN has 3 pre registered dependencies:
  - <code>app</code> Makes the express app available .
  - <code>auth</code> Includes some basic authentication functions
  - <code>database</code> Contains the Mongoose database connection</p>

<blockquote>
  <p>All dependencies specified must be registered in order to use them</p>

<div class="pilwrap" id="dependency%20injection">
  <h3>
    <a href="#dependency%20injection" name="dependency%20injection" class="pilcrow">&#182;</a>
    Dependency Injection
  </h3>
</div>


<blockquote>
  <p>An injection is the passing of a dependency (a service) to a dependent
  object (a client). The service is made part of the client's state.
  Passing the service to the client, rather than allowing a client to
  build or find the service, is the fundamental requirement of the
  pattern. <a href="http://en.wikipedia.org/wiki/Dependency_injection">Wikipedia</a></p>
</blockquote>

<p>Dependency injection allows you to declare what dependencies you require and rely on the package system to resolve all dependencies for you. Any package registered is automatically made available to anyone who would like to depend on them.</p>

<p>Looking again at the registration example we can see that <code>MyPackage</code> depends on the <code>Tokens</code> package and can make use of its full functionality, including overriding it.</p>


<div class="highlight"><pre><code><span class="c1">// Example of registering the tokens package</span>
<span class="nx">MyPackage</span><span class="p">.</span><span class="nx">register</span><span class="p">(</span><span class="kd">function</span><span class="p">(</span><span class="nx">app</span><span class="p">,</span> <span class="nx">auth</span><span class="p">,</span> <span class="nx">database</span><span class="p">,</span> <span class="nx">Tokens</span><span class="p">)</span> <span class="p">{</span>

  <span class="c1">// I can make use of the tokens within my module</span>
  <span class="nx">MyPackage</span><span class="p">.</span><span class="nx">someExampleFunction</span><span class="p">(</span><span class="s1">&#39;some parameter&#39;</span><span class="p">);</span>

  <span class="c1">// I can override functions</span>
  <span class="nx">MyPackage</span><span class="p">.</span><span class="nx">someExampleFunction</span> <span class="o">=</span> <span class="kd">function</span><span class="p">(</span><span class="nx">param</span><span class="p">)</span> <span class="p">{</span>
    <span class="c1">//my custom logic goes here</span>
  <span class="p">};</span>
<span class="p">});</span>
</code></pre></div>



<blockquote>
  <p>Packages when in code are used in a capitalized form</p>
</blockquote>

<div class="pilwrap" id="assets%20and%20aggregation">
  <h3>
    <a href="#assets%20and%20aggregation" name="assets%20and%20aggregation" class="pilcrow">&#182;</a>
    Assets and Aggregation
  </h3>
</div>


<p>All assets such as images, javascript libraries and CSS stylesheets should be within <code>public/assets</code> of the package file structure.</p>

<p>Javascript and CSS from <code>assets</code> can be aggregated to the global aggregation files. By default all javascript is automatically wrapped within an anonymous function unless given the option <code>{global:true}</code> to not enclose the javascript within a contained scope</p>


<div class="highlight"><pre><code><span class="c1">//Adding jquery to the mean project</span>
<span class="nx">MyPackage</span><span class="p">.</span><span class="nx">aggregateAsset</span><span class="p">(</span><span class="s1">&#39;js&#39;</span><span class="p">,</span><span class="s1">&#39;jquery.min.js&#39;</span><span class="p">);</span>

<span class="c1">//Adding another library - global by default is false</span>
<span class="nx">MyPackage</span><span class="p">.</span><span class="nx">aggregateAsset</span><span class="p">(</span><span class="s1">&#39;js&#39;</span><span class="p">,</span><span class="s1">&#39;jquery.min.js&#39;</span><span class="p">,</span> <span class="p">{</span><span class="nx">global</span><span class="o">:</span><span class="kc">true</span><span class="p">});</span>

<span class="c1">//Adding some css to the mean project</span>
<span class="nx">MyPackage</span><span class="p">.</span><span class="nx">aggregateAsset</span><span class="p">(</span><span class="s1">&#39;css&#39;</span><span class="p">,</span><span class="s1">&#39;default.css&#39;</span><span class="p">);</span>
</code></pre></div>



<blockquote>
  <p>Javascript files outside of assets are automatically aggregated and
  injected into the mean project. As a result libraries that you do not
  want aggregated should be placed within <code>public/assets/js</code></p>
</blockquote>

<p>The aggregation supports the ability to control the location of where to inject the aggregated code and if you add a weight and a group to your aggregateAsset method you can make sure it's included in the correct region.</p>


<div class="highlight"><pre><code><span class="nx">MyPackage</span><span class="p">.</span><span class="nx">aggregateAsset</span><span class="p">(</span><span class="s1">&#39;js&#39;</span><span class="p">,</span><span class="s1">&#39;first.js&#39;</span><span class="p">,{</span><span class="nx">global</span><span class="o">:</span><span class="kc">true</span><span class="p">,</span>  <span class="nx">weight</span><span class="o">:</span> <span class="o">-</span><span class="mi">4</span><span class="p">,</span> <span class="nx">group</span><span class="o">:</span> <span class="s1">&#39;header&#39;</span><span class="p">});</span>
</code></pre></div>



<blockquote>
  <p>The line that gets loaded in your head.html calls the header group and injects the js you want to include first-
  in packages/system/server/views/includes/head.html </p>
  
  <script type="text/javascript" src="/modules/aggregated.js?group=header"></script>
</blockquote>

<span class="p">});</span>

<span class="c1">// Another save settings example this time with no callback</span>
<span class="c1">// This writes over the last settings.</span>

    <a href="#overriding%20the%20default%20layouts" name="overriding%20the%20default%20layouts" class="pilcrow">&#182;</a>
    Overriding the default layouts
  </h3>
</div>


<p>One is able to override the default layout of the application through a custom package.</p>

<p>Below is an example overriding the default layout of system and instead using the layourts found locally within the package</p>


<div class="highlight"><pre><code><span class="nx">MyPackage</span><span class="p">.</span><span class="nx">register</span><span class="p">(</span><span class="kd">function</span><span class="p">(</span><span class="nx">system</span><span class="p">,</span> <span class="nx">app</span><span class="p">)</span> <span class="p">{</span>
  <span class="nx">app</span><span class="p">.</span><span class="nx">set</span><span class="p">(</span><span class="s1">&#39;views&#39;</span><span class="p">,</span> <span class="nx">__dirname</span> <span class="o">+</span> <span class="s1">&#39;/server/views&#39;</span><span class="p">);</span>
  <span class="c1">// ...</span>
</code></pre></div>



<blockquote>
  <p>Please note that the package must depend on <code>System</code> to ensure it is
  evaluated after <code>System</code> and can thus override the views folder</p>
</blockquote>


<div class="pilwrap" id="overriding%20views">
  <h3>
    <a href="#overriding%20views" name="overriding%20views" class="pilcrow">&#182;</a>
    Overriding views
  </h3>
</div>


<p>You may override public views used by certain core packages.  To create a custom home page, you would create a custom package and modify the script in it's public folder like so:</p>


<div class="highlight"><pre><code><span class="nx">angular</span><span class="p">.</span><span class="nx">module</span><span class="p">(</span><span class="s1">&#39;mean.mycustompackage&#39;</span><span class="p">,</span> <span class="p">[</span><span class="s1">&#39;mean.system&#39;</span><span class="p">])</span>
<span class="p">.</span><span class="nx">config</span><span class="p">([</span><span class="s1">&#39;$viewPathProvider&#39;</span><span class="p">,</span> <span class="kd">function</span><span class="p">(</span><span class="nx">$viewPathProvider</span><span class="p">)</span> <span class="p">{</span>
  <span class="nx">$viewPathProvider</span><span class="p">.</span><span class="nx">override</span><span class="p">(</span><span class="s1">&#39;system/views/index.html&#39;</span><span class="p">,</span> <span class="s1">&#39;mycustompackage/views/myhomepage.html&#39;</span><span class="p">);</span>
<span class="p">}]);</span>
</code></pre></div>



<p>This will render <em>mycustompackage/views/myhomepage.html</em> as the home page.</p>

<div class="pilwrap" id="config">
  <h2>
    <a href="#config" name="config" class="pilcrow">&#182;</a>
    Config
  </h2>
</div>


<p>All the configuration is specified in the <a href="/config/">config</a> folder,
through the <a href="config/env/">env</a> files, and is orchestrated through the <a href="https://github.com/linnovate/meanio">meanio</a> NPM module.
Here you will need to specify your application name, database name, and hook up any social app keys if you want integration with Twitter, Facebook, GitHub, or Google.</p>


<div class="pilwrap" id="environmental%20settings">
  <h3>
    <a href="#environmental%20settings" name="environmental%20settings" class="pilcrow">&#182;</a>
    Environmental Settings
  </h3>
</div>


<p>There is a shared environment config: <strong>all</strong></p>

<ul>
<li><strong>root</strong> - This the default root path for the application.</li>
<li><strong>port</strong> - DEPRECATED to <strong>http.port</strong> or <strong>https.port</strong>.</li>
<li><strong>http.port</strong> - This sets the default application port.</li>
<li><strong>https</strong> - These settings are for running HTTPS / SSL for a secure application.</li>
<li><strong>port</strong> - This sets the default application port for HTTPS / SSL. If HTTPS is not used then is value is to be set to <strong>false</strong> which is the default setting. If HTTPS is to be used the standard HTTPS port is <strong>443</strong>.</li>
<li><strong>ssl.key</strong> - The path to public key.</li>
<li><strong>ssl.cert</strong> - The path to certificate.</li>
</ul>

<p>There are three environments provided by default: <strong>development</strong>, <strong>test</strong>, and <strong>production</strong>.
Each of these environments has the following configuration options:</p>

<ul>
<li><strong>db</strong> - This is where you specify the MongoDB / Mongoose settings</li>
<li><strong>url</strong> - This is the url/name of the MongoDB database to use, and is set by default to <strong>mean-dev</strong> for the development environment.</li>
<li><strong>debug</strong> - Setting this option to <strong>true</strong> will log the output all Mongoose executed collection methods to your console.  The default is set to <strong>true</strong> for the development environment.</li>
<li><strong>options</strong> - These are the database options that will be passed directly to mongoose.connect in the <strong>production</strong> environment: [server, replset, user, pass, auth, mongos] (<a href='http://mongoosejs.com/docs/connections.html#options'>http://mongoosejs.com/docs/connections.html#options</a>) or read [this] (<a href='http://mongodb.github.io/node-mongodb-native/driver-articles/mongoclient.html#mongoclient-connect-options'>http://mongodb.github.io/node-mongodb-native/driver-articles/mongoclient.html#mongoclient-connect-options</a>) for more information.</li>
<li><strong>app.name</strong> - This is the name of your app or website, and can be different for each environment. You can tell which environment you are running by looking at the TITLE attribute that your app generates.</li>
<li><strong>Social OAuth Keys</strong> - Facebook, GitHub, Google, Twitter. You can specify your own social application keys here for each platform:
<ul><li><strong>clientID</strong></li>
<li><strong>clientSecret</strong></li>
<li><strong>callbackURL</strong></li></ul></li>
<li><strong>emailFrom</strong> - This is the from email address displayed when sending an email.</li>
<li><strong>mailer</strong> - This is where you enter your email service provider, username and password.</li>
</ul>

<p>To run with a different environment, just specify NODE_ENV as you call grunt:</p>


<div class="highlight"><pre><code><span class="nv">$ NODE_ENV</span><span class="o">=</span><span class="nb">test </span>grunt
</code></pre></div>



<p>If you are using node instead of grunt, it is very similar:</p>


<div class="highlight"><pre><code><span class="nv">$ NODE_ENV</span><span class="o">=</span><span class="nb">test </span>node server
</code></pre></div>



<p>To simply run tests</p>


<div class="highlight"><pre><code><span class="nv">$ </span>npm <span class="nb">test</span>
</code></pre></div>



<blockquote>
  <p>NOTE: Running Node.js applications in the <strong>production</strong> environment enables caching, which is disabled by default in all other environments.</p>
</blockquote>

<div class="pilwrap" id="staying%20up%20to%20date">
  <h2>
    <a href="#staying%20up%20to%20date" name="staying%20up%20to%20date" class="pilcrow">&#182;</a>
    Staying up to date
  </h2>
</div>


<p>After initializing a project, you'll see that the root directory of your project is already a git repository. MEAN uses git to download and update its own code. To handle its own operations, MEAN creates a remote called <code>upstream</code>. This way you can use git as you would in any other project. </p>

<p>To update your MEAN app to the latest version of MEAN</p>


<div class="highlight"><pre><code><span class="nv">$ </span>git pull upstream master
<span class="nv">$ </span>npm install
</code></pre></div>



<p>To maintain your own public or private repository, add your repository as remote. See here for information on <a href="https://help.github.com/articles/adding-an-existing-project-to-github-using-the-command-line">adding an existing project to GitHub</a>.</p>


<div class="highlight"><pre><code><span class="nv">$ </span>git remote add origin &lt;remote repository URL&gt;
<span class="nv">$ </span>git push -u origin master
</code></pre></div>




<div class="pilwrap" id="hosting%20mean">
  <h2>
    <a href="#hosting%20mean" name="hosting%20mean" class="pilcrow">&#182;</a>
    Hosting MEAN
  </h2>
</div>


<p>Since version 0.4.2 MEAN provides a command to easily upload your app to the <em>mean cloud</em>.
To do so all you need to do is the following steps.</p>

<ol>
<li>make sure you have a unique name for your app (not the default mean) and that the name is in the package.json</li>
<li>Run <code>mean deploy</code></li>
<li>It will create the meanio remote which can be used to update your remote app by <code>git push meanio master</code></li>
<li>You can add remote command using the --remote flag for instance to add a role to a user on the remote cloud instance run <code>mean user -a RoleName emailAddress --remote</code></li>
<li>To get an idea of whats happening on the mean log (node.js based logging) run <code>mean logs -n 100</code> to get the last 100 lines...</li>
</ol>

<div class="pilwrap" id="openshift">
  <h3>
    <a href="#openshift" name="openshift" class="pilcrow">&#182;</a>
    OpenShift
  </h3>
</div>


<ol>
<li>Register for an account on Openshift (<a href='http://www.openshift.com'>http://www.openshift.com</a>).</li>
<li>Create an app on Openshift by choosing a 'Node' type site to create. Create the site by making Openshift use Linnovate's Openshift git repo as its source code (<a href='https://github.com/linnovate/mean-on-openshift.git'>https://github.com/linnovate/mean-on-openshift.git</a>).</li>
<li>On the second screen after the new application has been created, add a Mongo database.</li>
<li>When the site has been built, you can visit it on your newly created domain, which will look like my-domain.openshift.com. You may need to restart the instance on Openshift before you can see it. It will look like Mean.io boilerplate.</li>
<li>On your new app's console page on Openshift, make a note of the git repo where the code lives. Clone that repo to your local computer where your mean.io app codebase is.</li>
<li>Merge your completed local app into this new repo. You will have some conflicts, so merge carefully, line by line.</li>
<li>Commit and push the repo with the Openshift code back up to Openshift. Restart your instance on Openshift, you should see your site!</li>
</ol>
